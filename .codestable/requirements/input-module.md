# 运行时输入动作管理

## 用户故事

- 作为玩法开发者，我希望读取“跳跃”“攻击”“打开背包”这类动作，而不是在各处直接判断具体按键。
- 作为需要支持键鼠和手柄的人，我希望一个动作可以绑定多个输入来源，而不是为每种设备写一套分支。
- 作为 UI 或调试工具开发者，我希望能查询动作当前是否按下、这一帧是否触发，而不是自己维护上一帧状态。
- 作为框架维护者，我希望输入轮询、动作状态、启用禁用和清理有统一规则，而不是散落在 Procedure、Combat、UI 和场景脚本里。

## 为什么需要

玩家输入会穿过很多系统：菜单、战斗、调试面板、剧情交互和相机控制都可能读取同一组按键。没有输入层时，底层按键判断会到处出现，后续改键位、禁用某个输入场景或支持手柄时，每个业务点都要改。

## 怎么解决

提供一个运行时输入动作能力：业务先登记动作和绑定，运行时统一轮询输入并生成每帧动作状态。调用方读取动作名即可知道是否按下、刚按下、刚松开或当前数值；不同场景可以启用或禁用自己的动作组。

## 边界

- 它不负责设计具体玩法操作方案，也不决定默认键位是否好用。
- 它不替代 UI 窗口系统；UI 焦点、按钮导航和界面交互仍由 UI 层决定是否接入。
- 它不做复杂手势识别、连招判定、输入缓冲、录像回放或网络同步。
- 首版不承诺多玩家本地输入、设备热插拔策略或用户自定义键位持久化。
- 首版不强制引入新的 Unity 输入包；如果以后要切换底层输入系统，应通过适配层演进。
