# 可撤销命令历史

## 用户故事

- 作为玩法或工具开发者，我希望把一次编辑、建造或状态修改封装成可撤销命令，而不是每个系统自己写一套回退逻辑。
- 作为运行时编辑功能的使用者，我希望误操作后可以撤销和重做，而不是只能重新调整场景或数据。
- 作为维护框架的人，我希望命令执行、撤销栈和重做栈有统一规则，而不是不同模块对历史记录的清理时机不一致。
- 作为调试复杂交互的人，我希望能看到当前是否可以撤销或重做，而不是靠业务侧猜测历史栈状态。

## 为什么需要

很多游戏内编辑、调试工具和可交互系统都会遇到“做了某件事后想退回”的问题。没有统一能力时，每个业务点都会自己记录旧值、清重做栈、处理失败回滚，最后容易出现撤销顺序错乱、重复撤销、重做被旧操作污染和模块关闭后状态残留。

## 怎么解决

提供一个命令历史能力：调用方把一次可执行操作交给系统，系统负责执行它、记录历史、判断能否撤销和重做，并在新命令执行后清理过期的重做历史。需要撤销或重做时，系统按栈顺序调用对应操作，并把历史状态更新到一致结果。

## 边界

- 它不负责设计具体业务命令，也不替业务保存完整世界快照。
- 它不替代操作模块；长耗时资源加载、下载和异步工作流仍走已有操作体系。
- 它不提供编辑器菜单、控制台命令、脚本解释器或网络指令分发。
- 首版不覆盖跨帧事务、多人同步、持久化历史、热重载恢复或分支历史浏览。
- 命令能否安全撤销由命令实现者保证；系统只统一调用顺序、历史栈和错误语义。
