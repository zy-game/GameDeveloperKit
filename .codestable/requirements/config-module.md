# 运行时配置表读取与查询

## 用户故事

- 作为玩法开发者，我希望用同一套方式读取道具、关卡、数值等配置表，而不是每种文件格式都写一套解析代码。
- 作为配置维护者，我希望配置表可以来自 XML、CSV、JSON 或 Unity 可编辑资产，而不是被迫迁移到单一格式。
- 作为框架维护者，我希望配置加载、缓存、重复 key、缺字段和解析失败都有统一规则，而不是错误散在业务代码里。

## 为什么需要

游戏业务会频繁读取静态配置。没有统一能力时，配置来源、解析方式、缓存和错误处理会分散在各个系统里，后续改格式或排查配置问题都很痛苦。

## 怎么解决

提供一个运行时配置能力：用户按配置表名称加载表，系统根据来源选择对应解析方式，把数据转成可查询的表对象，并统一缓存和校验。业务只关心“我要哪张表、查哪个 key”，不用关心底层来自哪种格式。

## 边界

- 它不负责编辑配置表内容，也不提供可视化配置编辑器。
- 它不负责热更新发布、远端下载、版本比对或差量补丁。
- 它不替代资源模块；配置文件或资产仍需要先能被项目资源体系或 Unity 资源路径找到。
- 它默认处理静态表格配置，不解决运行中玩家存档、服务器下发状态或频繁写入的数据。
- 它要求每张需要按 key 查询的表有稳定主键；没有主键的列表型配置只能按列表读取。
