# 场景框架启动入口

## 用户故事

- 作为接入框架的业务开发者，我希望在场景里挂一个启动入口就能进入指定业务流程，而不是把 App 启动、模块准备和流程切换散写在临时脚本里。
- 作为示例或测试场景维护者，我希望能在 Inspector 里选择初始化完成后进入哪个流程，而不是为了每个场景写一个专用 bootstrap。
- 作为框架维护者，我希望资源、声音、配置和数据的准备边界清楚，而不是让播放层、UI 或剧情 demo 顺手承担启动编排。
- 作为调试者，我希望启动失败、重复启动和销毁关闭都有统一可观察结果，而不是静默吞错或重复进入流程。

## 为什么需要

框架已经把模块生命周期收敛成同步外壳，资源、声音、配置、数据和流程切换都需要由业务入口显式编排。如果每个场景都自己写一份启动脚本，很容易把资源初始化、Loading UI、剧情播放和 Procedure 切换揉在一起，后续示例和业务场景会越来越难复用。

## 怎么解决

提供一个可挂载的场景启动组件：用户在 Inspector 中选择目标流程，并按需要开启资源初始化、配置/数据外壳解析和声音配置。启动时组件先进入框架生命周期，再准备这些被声明的模块，最后切换到目标流程；销毁时也可以按配置关闭框架。业务仍然可以在自定义流程里手动调用同一套模块 API。

## 边界

- 它不自动扫描并启动所有模块；只准备 Inspector 中显式声明的模块能力。
- 它不替代具体业务流程；初始化完成后只切换到用户选择的目标流程。
- 它不负责创建业务场景、加载首个游戏场景、注册剧情、播放剧情或打开 Loading UI。
- 它不把资源、声音配置做成全局默认资产；配置来自启动组件或调用方显式传入。
- 它不负责编辑器安装向导、菜单项或自动把启动组件塞进所有场景。
- 它不解决第三方插件、业务自定义模块或服务器生命周期管理。
