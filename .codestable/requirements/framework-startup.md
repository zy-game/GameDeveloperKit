# 框架统一启动与关闭

## 用户故事

- 作为接入框架的业务开发者，我希望在场景里放一个启动入口就能启动整套框架，而不是每个项目都手写一遍模块注册顺序。
- 作为框架维护者，我希望模块按依赖关系启动和反向关闭，而不是靠使用者猜哪个模块该先注册。
- 作为测试或示例场景维护者，我希望启动失败和重复启动有统一结果，而不是因为多个场景对象或手动注册造成状态不一致。

## 为什么需要

框架模块已经越来越多，资源、下载、文件、配置、UI、声音、调试等能力之间也有依赖。如果每个业务场景都自己排列注册顺序，很容易出现资源模块启动时操作模块还没准备好、关闭时资源还没释放完就清掉底层模块这类问题。统一启动能力可以把这条依赖链收束到框架自己维护。

## 怎么解决

提供一个运行时启动入口：用户把启动组件放进场景，框架就按固定依赖顺序启动需要的模块；场景销毁或应用退出时，框架按相反顺序关闭模块并清理状态。业务也可以在测试或自定义引导流程里手动触发同一套启动和关闭能力。

## 边界

- 它不负责创建业务场景、加载首个游戏场景或切换流程。
- 它不自动扫描所有程序集里的模块；默认启动哪些框架能力需要由框架显式维护。
- 它不替代单个模块自己的配置、资源和错误处理规则。
- 它不负责编辑器安装向导、菜单项或自动把启动组件塞进所有场景。
- 它不解决第三方插件、业务自定义模块或服务器生命周期管理。
