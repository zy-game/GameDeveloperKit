# 战斗 ECS 编排模块

## 用户故事

- 作为战斗开发者，我希望用统一的 World、Entity、Component 和 System 表达战斗对象与逻辑，而不是每个玩法都直接操作底层 ECS 库。
- 作为框架维护者，我希望 combat 层能包住 massive 的世界与回滚能力，并提供项目自己的命名和生命周期约束，而不是把第三方 API 泄漏到所有业务代码里。
- 作为写战斗系统的人，我希望系统只处理满足 include / exclude 条件的实体，并在实体进入、离开和每帧更新时收到一致回调。
- 作为调试战斗问题的人，我希望实体创建、销毁、组件变化和系统更新有统一入口，而不是散落在不同脚本和集合里。

## 为什么需要

战斗逻辑通常会同时处理大量实体、状态组件和按条件运行的系统。直接把底层 ECS API 暴露给业务虽然快，但会让实体生命周期、组件筛选、系统回调和回滚边界分散在各处。随着战斗系统变多，谁负责创建实体、组件变化后系统何时感知、系统 update 顺序如何保证，都会变成隐性约定。

## 怎么解决

提供一个运行时 `CombatModule`，模块持有默认战斗世界；战斗世界封装底层 ECS 世界，并集中持有实体管理器和系统管理器。业务通过实体管理器创建 / 销毁实体、添加 / 移除组件，通过系统管理器注册系统；系统声明 include / exclude 组件条件后，只会收到符合条件实体的 create、destroy 和 update 回调。底层 ECS 库仍负责数据存储、查询与回滚，combat 层负责项目级生命周期和 API 口径。

## 边界

- 它不实现具体战斗规则、伤害公式、技能系统、AI、寻路、动画、特效或网络同步。
- 它不替代 massive；底层实体、组件存储、查询和回滚仍由 massive 承担，combat 层只做二次封装。
- 它不负责 Unity GameObject / Transform 同步；需要表现层同步时由业务系统显式处理。
- 它不提供多线程或 Job System 安全承诺；首版公开 API 假定 Unity 主线程调用。
- 它不自动扫描程序集注册所有系统；系统注册顺序由调用方显式控制。
- 它不把 CombatModule 纳入框架默认启动计划；需要战斗 ECS 的场景或流程按需注册。
