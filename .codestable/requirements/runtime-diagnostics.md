# 运行时诊断工具集

## 用户故事

- 作为测试或 QA，我希望在游戏运行时直接看到日志、帧率、内存和设备指标，而不是每次都连 Editor 或翻系统日志。
- 作为线上问题排查者，我希望能让玩家或测试包上传最近日志和崩溃线索，而不是靠截图和口头描述还原问题。
- 作为业务开发者，我希望埋点和调试命令有统一入口，而不是每个系统自己写一套临时按钮、作弊码或上报代码。
- 作为框架维护者，我希望日志模块能升级成清晰的调试模块，而不是另外并行出一套诊断模块入口。

## 为什么需要

当前日志模块只负责把结构化日志转发到输出目标，适合作为调试基础，但还不能覆盖真实运行时排查：看不到最近日志列表，不能主动上传调试包，埋点和调试命令缺少统一入口，崩溃后也没有稳定的本地线索收集。随着模块增多，问题会分散在资源、下载、配置、UI 和业务链路里，需要把日志模块升级成运行时调试入口，把这些线索收束起来。

## 怎么解决

提供一个运行时调试能力：系统持续收集最近日志和关键指标，用 IMGUI 按需展示调试面板，允许业务记录埋点事件，支持把本地调试包上传到外部接收方，并让命令模块支持通过命令名和参数执行调试命令。各能力通过独立通道、采样器、面板和命令扩展，默认只在开发或测试环境启用。

## 边界

- 它不新增独立 DiagnosticsModule；现有 Logger 能力升级为 Debug/调试入口。
- 它不承诺首版接入某个固定云平台、崩溃 SDK 或商业 analytics 服务。
- 它不使用 UGUI 或业务 UI 模块绘制调试界面；运行时面板只使用 IMGUI。
- 它不在调试模块里另建 GM 命令系统；命令名和参数执行能力属于 CommandModule。
- 它不默认收集隐私数据、账号凭证、支付信息或业务敏感字段。
- 它不作为玩家正式 UI；运行时面板默认面向开发、QA、测试包或明确开启的诊断场景。
