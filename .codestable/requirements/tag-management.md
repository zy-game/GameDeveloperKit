# 项目标签目录管理

## 用户故事

- 作为资源或工具开发者，我希望在一个编辑器窗口里维护项目自定义标签，而不是把标签名写在不同脚本、资源和文档里。
- 作为资源维护者，我希望能看到当前项目已有的资源标签，并把它们和项目自定义标签放在同一套目录里管理。
- 作为玩法开发者，我希望运行时能通过配置模块读取标签目录，而不是手写字符串常量或依赖 Editor API。
- 作为框架维护者，我希望 Unity 自带标签、资源标签和项目自定义标签边界清楚，而不是不同来源混在一起难以追踪。

## 为什么需要

标签通常会同时出现在资源收集、编辑器工具、运行时筛选和业务判断里。没有统一目录时，标签会散落在 AssetDatabase、Unity TagManager、manifest label 和手写字符串中，后续改名、排查缺失或确认某个标签是否还能用都很费劲。

## 怎么解决

提供一个项目级标签目录能力：编辑期用可视化窗口维护自定义标签，并把可读取的 Unity 资源标签和 Unity 标签同步成固定分组；运行时读取保存好的标签目录，业务只面对同一份只读标签配置。

## 边界

- 它不负责批量给资源打标签，也不替代 Unity Inspector 或现有资源编辑器里的单资源标签操作。
- 它不直接编辑 Unity TagManager；Unity 标签只作为可读取来源同步进目录。
- 它不改变资源 manifest 的标签字段含义，也不改变资源模块加载语义。
- 它不提供运行时写回、远端热更新、版本差量或玩家存档标签。
- 它要求标签目录先由编辑器保存到项目资源中，运行时只读取这份固化结果。
