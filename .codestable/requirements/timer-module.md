# 运行时时钟与调度

## 用户故事

- 作为玩法开发者，我希望能登记倒计时、延时执行和循环计时任务，而不是每个系统自己写 `MonoBehaviour`、协程或临时累加器。
- 作为框架模块维护者，我希望资源、调试、声音、流程等模块共享清晰的运行时时钟，而不是各自猜当前帧时间。
- 作为 QA 或调试工具使用者，我希望能看到当前 tick、时间缩放和已登记计时任务，而不是只能靠日志推断某个延迟是否还活着。

## 为什么需要

游戏里大量逻辑都和时间有关：倒计时、延迟执行、循环采样、冷却、超时、动画外的轮询、调试面板刷新。没有统一能力时，代码会散落成很多小计时器，取消和关闭时也容易漏，排查“为什么晚了一帧”或“为什么还在回调”会很痛苦。

## 怎么解决

提供一个共享的运行时时间能力：系统统一维护当前 tick、累计时间和每帧增量，并让调用方登记可取消、可暂停、可按所有者清理的倒计时、延时执行或循环任务。调试工具可以读取同一份时间状态和任务快照，帮助定位时间相关问题。

## 边界

- 它不替代 Unity 协程、UniTask 或业务异步流程，只负责时间驱动的简单调度。
- 它不提供后台线程、任务队列、优先级、重试或作业系统。
- 它不承诺网络回滚、锁步战斗或物理模拟级别的确定性。
- 它不管理真实世界日期、时区、服务器时间或活动日历。
