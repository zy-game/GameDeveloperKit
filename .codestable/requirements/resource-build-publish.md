# 构建资源包与发布资源版本

## 用户故事

- 作为维护热更新资源的开发者，我希望配置好 package 和 bundle 后直接构建出可运行的资源包，而不是手写清单再单独跑打包脚本。
- 作为需要发布远端资源的人，我希望构建完成后在独立 Publisher 工具里选择平台、渠道、区域和存储桶，再上传或回滚资源版本，而不是把上传配置混在资源编辑器里。
- 作为维护多个对象存储平台的人，我希望输入对应平台的 appid / key 后能查看区域列表、存储桶列表，并能创建或删除存储桶。
- 作为排查资源问题的人，我希望构建结果能明确告诉我生成了哪些 bundle、manifest 和失败原因，而不是只看到一串控制台日志。

## 为什么需要

资源编辑器已经能维护收集规则，但从配置到真正可被运行时加载的资源包之间还缺一段稳定构建链路。远端上传、渠道配置、对象存储桶管理和版本回滚是发布域职责，继续放进资源编辑器会让资源编辑器变成混合工具，后续支持多个平台时更难维护。

## 怎么解决

资源编辑器只负责检查、生成构建计划、构建本地资源包、写出运行时 manifest 和构建报告。新增 Resource Publisher 工具读取构建报告，管理平台凭证引用、区域、存储桶、渠道、版本记录，并执行上传、更新、回滚和存储桶管理操作。构建失败停留在资源编辑器；发布失败停留在 Publisher，不互相隐式重试。

## 边界

- 它不负责运行时下载缓存、差量补丁、灰度发布或 CDN 刷新。
- 它不在资源编辑器里保存云服务密钥、区域、存储桶或渠道配置；这些属于 Resource Publisher。
- Resource Publisher 可以保存平台账号的非密钥配置和凭证引用，但 appid / key 的具体值由用户提供或来自本机安全来源，不由工具计算。
- 它不替代资源编辑器的收集规则管理；没有配置好的 package / bundle 不能构建。
- 它不负责重设计运行时 manifest 字段；构建产物必须继续匹配当前资源模块能读取的清单格式。
- 它不把对象存储平台 SDK 暴露给 Runtime；对象存储管理和上传都在 Editor-only 工具中。
