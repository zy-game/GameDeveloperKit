# 运行时 UI 窗口管理

## 用户故事

- 作为玩法或业务开发者，我希望用窗口类型打开和关闭 UI，而不是在每个业务点手写 prefab 加载、实例化和销毁。
- 作为 UI 开发者，我希望在 Inspector 中选择需要绑定的 Image、Button、Text 等组件，并生成强类型绑定代码，而不是运行时手写 `GetComponent`。
- 作为 UI 开发者，我希望窗口在 `OnAwake` 时一次性缓存全部绑定组件到 `ExampleModel`，后续逻辑直接读字段，而不是每次访问都重新查找。
- 作为做页面流程的人，我希望主界面、弹窗、加载层、提示层有明确层级、安全区和回退规则，而不是靠场景层级临时调整。
- 作为移动端适配的人，我希望背景可以铺满刘海屏和挖孔屏，但按钮、文本等关键内容保持在安全区内。
- 作为维护框架的人，我希望 UI 资源跟窗口生命周期一起释放，而不是打开几轮界面后留下隐藏对象或资源句柄。

## 为什么需要

游戏运行时 UI 通常同时涉及资源加载、Canvas 层级、安全区、窗口栈、组件绑定和关闭释放。没有统一能力时，每个页面都会重复处理这些细节，最容易出现层级错乱、刘海屏遮挡、重复打开、返回逻辑不一致和资源没释放。

## 怎么解决

提供一个运行时 UI 窗口管理能力：开发者把窗口 prefab 和窗口逻辑关联起来，运行时按窗口类型打开、切换、回退和关闭。系统统一负责加载界面、放入正确层级、把内容约束到安全区、生成组件绑定代码，并在 `OnAwake` 时一次性把组件缓存进模型，后续读取不再重复查找；同时维护窗口状态，并在关闭或模块退出时清理对象和资源。

## 边界

- 它不负责制作 UI prefab、布局美术、动画效果或交互视觉规范。
- 它不替代资源模块；UI 资源仍需要先能被项目资源体系加载。
- 它只提供 UIDocument 绑定 Inspector 和代码生成，不做完整 UI 设计器，也不属于资源编辑器功能。
- 它不提供登录、背包、设置等具体业务页面，只提供业务页面挂载和生命周期规则。
- 首版不覆盖多实例同类型窗口、复杂转场动画、焦点导航、国际化和输入系统适配。
