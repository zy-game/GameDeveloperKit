# 运行时日志统一管理

> 2026-06-05：独立 Logger 能力不再继续推进。日志能力归入 `runtime-diagnostics` 的 Debug 调试中枢，后续不保留 `LoggerModule` / `App.Logger` 入口。

## 用户故事

- 作为框架模块开发者，我希望所有模块都通过同一个日志入口记录诊断信息，而不是每个模块自己决定怎么 `Debug.Log`。
- 作为定位线上问题的人，我希望日志能按等级和分类筛选，而不是在一大串无关输出里手动翻。
- 作为接入项目的业务开发者，我希望能切换日志输出方式，而不是为了接入文件、Unity Console 或自定义上报到处改调用点。

## 为什么需要

框架模块会越来越多，问题也会分散在资源加载、下载、事件、UI、操作等链路里。没有统一日志能力时，日志格式、等级、分类和输出去向都靠各模块各自处理，排查问题时很难把同一次流程串起来，也很难控制正式包里哪些信息该输出。

## 怎么解决

提供一个统一日志能力：调用方只表达“这是什么级别、属于哪个分类、内容是什么”，系统负责按当前设置过滤，并把日志交给已启用的输出去向。默认能在开发期看到日志，后续也能替换或增加输出方式。

## 边界

- 它不负责修复错误，只负责记录和转发诊断信息。
- 它不把日志当作玩家可见提示；业务 UI 提示仍由业务/UI 模块处理。
- 它不默认上传远端日志；如果需要远端上报，应作为后续扩展明确设计。
- 它不接管第三方 SDK 内部日志，只管理通过框架日志入口写入的内容。

## 变更日志

- 2026-06-05：用户确认 Logger 不再保留，独立 Logger 愿景标记为 outdated；日志能力并入运行时 Debug 调试入口。
