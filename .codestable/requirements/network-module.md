# 统一网络连接和消息分发

## 用户故事

- 作为玩法或业务开发者，我希望通过统一入口建立 socket 连接并发送消息，而不是每个系统自己维护连接、序列化和等待响应。
- 作为登录、活动或配置接口的调用者，我希望普通 HTTP 请求也能走同一套错误分类和响应封装，而不是到处手写 UnityWebRequest。
- 作为框架维护者，我希望收到的网络消息能按消息类型分发给对象处理器或回调函数，而不是让业务模块自己维护一堆散落的 switch。
- 作为调试运行时问题的人，我希望网络连接状态、请求失败和消息分发有清晰可观察结果，而不是只看到某个回调没有触发。

## 为什么需要

联网玩法通常同时需要长连接和短请求：socket 负责实时消息，HTTP 负责登录、配置、活动和一次性 API。没有统一能力时，业务侧会重复写连接状态、收发等待、超时、回调注册、错误分类和关闭清理，问题排查也会分散在各个模块里。

## 怎么解决

提供一个运行时网络能力：系统统一管理网络连接、消息发送、响应等待、主动推送分发和 HTTP 请求封装。业务把消息对象交给连接发送，也可以按消息类型注册处理器或回调；连接关闭、请求失败和超时都用一致的结果暴露给调用方。

## 边界

- 它不负责文件下载、断点续传或大文件分片；这些仍属于 DownloadModule。
- 它不定义具体游戏业务协议、服务器地址发现、账号登录流程或加密握手细节。
- 它不做战斗同步、帧同步、状态回滚或可靠 UDP。
- 首版不承诺多线程安全；公开 API 默认在 Unity 主线程编排。
- 首版不把 Debug 日志自动发到服务端，只预留未来由 Network 读取已脱敏日志并发送的桥接位置。
