# 战斗同步与回滚

## 用户故事

- 作为做联网战斗的玩法开发者，我希望战斗能按统一帧推进和同步输入，而不是每个房间自己约定 Tick、延迟和补帧。
- 作为做客户端表现的人，我希望本地可以先预测战斗结果，并在权威状态回来后自动修正，而不是只靠重新进房或整局重开。
- 作为排查战斗不同步的人，我希望能看到哪一帧、哪个玩家输入、哪份状态校验出了问题，而不是只知道“客户端和服务器不一样”。
- 作为框架维护者，我希望回滚不只停留在底层 ECS 的世界复制，还能带上输入历史、状态帧和重演规则。

## 为什么需要

当前 Combat 已经能本地固定步推进，也能借助底层 ECS 做世界回滚，但这些能力只解决“本机怎么跑”。一旦进入联网战斗，还缺少统一的帧编号、输入同步、状态快照、状态校验和回滚重演口径。没有这层能力时，业务只能在 Combat 外再散写一套同步协议，最后很难保证不同玩法使用同一套规则。

## 怎么解决

让战斗运行时有一套面向联网场景的同步能力：战斗按帧消费输入，定期生成可校验的状态帧；收到远端输入或权威状态后，系统能判断是否需要回滚，并把历史输入重新跑到当前帧。业务仍然写自己的战斗系统和网络服务端，框架负责把帧、状态、校验和修正流程统一起来。

## 边界

- 它不实现匹配、房间、服务端战斗进程、鉴权、重连恢复或反作弊。
- 它不保证所有战斗逻辑天然确定性；随机数、浮点、时间读取和外部资源访问仍需要业务按同步规则约束。
- 它不自动同步所有组件字段；参与状态同步的实体和组件需要明确登记。
- 它不接管动画、特效、Transform 表现层或 GameObject 状态。
- 它不替代现有 Network 模块；网络模块仍只负责连接、消息收发和分发。
