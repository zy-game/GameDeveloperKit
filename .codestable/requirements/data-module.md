# 运行时数据保存与读取

## 用户故事

- 作为玩法开发者，我希望能按数据类型和 key 读取玩家档案、关卡进度或临时状态，而不是在每个系统里各自维护字典。
- 作为设置界面开发者，我希望音量、画质、引导状态这类本地数据能统一保存和恢复，而不是直接散落在 PlayerPrefs key 字符串里。
- 作为框架维护者，我希望运行时数据的创建、缓存、保存、版本回滚、清理和错误语义有统一规则，而不是每个模块自己决定怎么序列化。
- 作为调试存档问题的人，我希望能知道某份数据按什么类型和 key 保存，读取失败时能定位到具体数据槽。

## 为什么需要

运行中会变化的数据常常夹在配置、存档、设置和临时状态之间。没有统一入口时，业务会反复手写静态字段、字典、PlayerPrefs key 或文件读写，久了以后很难知道谁拥有某份数据、什么时候保存、丢失或格式错误时该怎么处理。

## 怎么解决

提供一个运行时数据中心：业务按数据类型和可选 key 获取同一份数据对象，系统负责缓存、区分默认实例和命名实例，并提供统一的保存、加载、版本回滚、删除和序列化入口。能被序列化的数据可以持久化到本地；不能安全序列化的数据仍可以作为运行期内存数据使用。

## 边界

- 它不替代静态配置表；只读数值表、标签目录和资源清单仍归配置或资源能力管理。
- 它不负责服务器同步、云存档、多端冲突合并或多人权威状态。
- 它不保存大型二进制资源、AssetBundle、贴图或音频；大文件和资源仍归文件或资源能力管理。
- 它不承诺自动序列化 Unity 场景对象、`UnityEngine.Object` 引用、委托、循环引用或复杂多态对象图。
- 它不提供编辑器存档查看器、加密、压缩、版本迁移 UI 或作弊防护。
- 它不自动侦测对象字段被直接修改；业务需要在合适时机显式保存。
