# 对象级无级状态机

## 用户故事

- 作为玩法开发者，我希望怪物、玩家、技能或交互物能按 Idle、Move、Attack、Dead 这类状态切换，而不是到处写 enum、bool 和 if 分支。
- 作为框架维护者，我希望对象级状态有统一的进入、离开、更新和释放顺序，而不是每个业务类自己约定一套生命周期。
- 作为排查战斗或 AI 问题的人，我希望能看到某个对象当前状态和上一次切换，而不是从散落日志里猜是哪段逻辑还在跑。
- 作为做顶层流程的人，我希望全局 Procedure 仍只管启动、登录、主菜单、战斗这类大阶段，小对象状态不要塞进 Procedure 里。

## 为什么需要

游戏里很多逻辑不是全局流程，但也天然是“同一时间只处在一个状态”：角色 AI、技能阶段、交互物开关、局部 UI 模式、Buff 生命周期。没有统一能力时，这些状态常被写成散落字段和手动回调，久了会出现重复进入、旧状态没清理、更新还在跑、以及不同对象各有一套叫法的问题。

## 怎么解决

提供一套对象级平铺状态机能力：每台状态机绑定一个业务对象，拥有一组平铺状态，同一时间只有一个当前状态。切换时统一走离开旧状态、进入新状态、更新当前状态和停止释放的生命周期；需要观察时可以查看当前状态和切换事件。顶层游戏阶段继续使用全局流程能力，局部对象状态用这套更轻的机制承载。

## 边界

- 它不替代全局流程状态机；启动、登录、主菜单、战斗等顶层阶段仍归 Procedure。
- 它不做行为树、GOAP、动画状态机、可视化状态图编辑器或技能编辑器。
- 它不做层级状态、并行状态、push / pop 状态栈、历史回退或状态持久化恢复。
- 它不负责加载资源、打开 UI、执行命令或调度计时任务；状态内部需要时调用已有模块。
- 它不承诺网络锁步、回滚确定性、多线程安全或 Burst / Job System 兼容。
